Report icons

Every report carries a generated emoji icon. Two independent flows produce them:

  1. Per-report icon — one emoji per Report, generated by a single LLM call against the bundled internal/icon prompt pinned to a specific agent (DeepSeek by default). Surfaces in the AI Reports hub, history rows, search hits, and the title bar of every report-scoped screen.
  2. Per-agent icon — one emoji per ReportAgent, generated by a 3-tier fallback chain against each agent's own (provider, model). Surfaces in the result page's icon row, the View → Icons grid, and the per-call cost tables.

Both flows are gated by master switches under Settings → AI Reports:

iconGenEnabled (default true)
Generate report icons — the per-report icon.
perModelIconGenEnabled (default true)
Generate per model icons — the per-agent 3-tier chain.

When either switch is off, the matching icon column doesn't generate; existing on-disk icons stay (re-enabling brings them back).

The four bundled prompts

Seeded from assets/prompts.json on every app start (delta merge — InternalPromptSeed only adds missing entries by (category, name), so user edits survive):

icon — DeepSeek
Substitutes @PROMPT@. Used by the per-report icon flow + alternative-icons fan-out.
report_icon — *N/A (any)
Substitutes @PROMPT@ + @RESPONSE@. Tier 2 of the per-agent chain + alternative-icons (when launched from the per-agent detail screen).
report_icon_chat — *N/A (any)
No substitutions — sent as third turn of a chat. Tier 1.
report_icon_3th — DeepSeek
Substitutes @RESPONSE@. Tier 3 fallback.

The bundled agent for icon / report_icon_3th is DeepSeek because it's cheap, fast, and reliably emits a single glyph. Users can re-pin via Settings → AI Setup → Internal prompts → icon (the dispatcher does a case-insensitive agent match so a manually registered Deepseek still resolves the bundled DeepSeek pin).

Per-report icon flow

ReportViewModel.kickOffIconGeneration runs at the start of generateGenericReports (the user just tapped Generate):

  1. Read the internal/icon prompt; bail if missing.
  2. Resolve the pinned agent — must exist in Settings.agents, match case-insensitively. Copy the agent with the effective API key (key lives on the Provider, not the Agent) and effective model (when the agent is pinned to a default-model alias).
  3. Substitute @PROMPT@ with the report's prompt text.
  4. Launch on viewModelScope.launch(Dispatchers.IO) — withTracerTags(reportId, category="Report icon") so the call's trace surfaces on the report's Trace screen.
  5. AnalysisRepository.analyzeWithAgent makes the call.
  6. Always normalise to exactly one emoji. extractFirstEmoji (data/EmojiExtract.kt) walks the response with BreakIterator.getCharacterInstance() and returns the first grapheme cluster whose lead codepoint sits in a known emoji block (Misc Symbols, Pictographs, Transport, Dingbats, Misc Technical, Regional Indicators, Mahjong / Domino / Playing cards, Supplemental Symbols). 200 OK with no emoji in the body → fallback 📝. Non-200 / network failure → Report.iconErrorMessage is set instead; the row renders ❌.
  7. Cost is computed from PricingCache.getPricing(provider, model) × (inputTokens, outputTokens) and persisted onto Report.iconInputCost / iconOutputCost (plus tokens).
  8. iconRefreshTick on UiState is bumped to force recomposition of every list and title bar reading the icon.

The call is launched on viewModelScope (AppViewModel's), not ReportViewModel's — so navigating away from the result screen doesn't cancel it.

Per-agent 3-tier chain

ReportViewModel.runReportIconsForAgent runs per agent as soon as that agent's primary call settles to SUCCESS — so a fast row can show its emoji before slower rows in the same report have finished generating.

Auto-fires when:

  • A fresh report's agent completes (via generateGenericReports).
  • A regenerated agent completes (via regenerateReport).

Tiers run sequentially; first one returning an extractable emoji wins:

Tier 1 — internal/report_icon_chat
Agent's own provider / model. Sent as the third turn of a user / assistant / user chat. The previous answer is replayed verbatim as the assistant turn.
Tier 2 — internal/report_icon
Agent's own. Substitutes @PROMPT@ + @RESPONSE@.
Tier 3 — internal/report_icon_3th
Fixed bundled DeepSeek. @RESPONSE@ only.

All three fail → the row's icon falls back to 📝 and iconWinningTier stays null.

Each call:

  • Pre-acquires a ProviderThrottle permit for the host and passes permitPreAcquired = true through the coroutine context element + TagPropagatingExecutor so the inline ProviderThrottleInterceptor doesn't double-count it.
  • Tags traces with withTracerTags(reportId, category="Report icons tier N (…)").
  • Bumps ReportAgent.iconInputCost / iconOutputCost by the call's spend regardless of whether the emoji extraction succeeded (the user paid either way).
  • Appends an IconCallRecord to Report.iconCalls so the export's per-call All tab can render every attempt — the failed earlier tiers AND the winning one — as its own row.
  • Posts to the global UsageStats ledger with kind="icon", attributed to the actual (provider, model) that ran (tier 3 uses DeepSeek even when the agent row itself uses a different model).

The Tier 1 path uses sendChat, which doesn't surface wire token counts — AppViewModel.estimateTokens (char-length heuristic) fills the gap, matching the dual-chat accounting.

Each job registers in reportIconsJobs under "$reportId|$agentId", so deleteReport's prefix sweep cancels them and a regenerate-fired re-run cancels the previous attempt for the same agent.

Find alternative icons

Bottom button on Agent icon detail (the per-agent emoji screen reached by tapping any icon-row glyph). Opens a model picker; the user picks any number of (provider, model) pairs.

startIconFanOut fans the bundled internal/icon prompt (when launched from the Reports hub-level alternative-icons button) or internal/report_icon with the dual @PROMPT@ + @RESPONSE@ substitution (when launched from the per-agent detail) across the picks:

  • Dedupes by "providerId:model".
  • Pre-populates IconCandidate.Running rows so the Alternative icons screen shows ⏳ for every pair the moment it opens, before any throttle permit is acquired.
  • One async per pick; each pre-acquires the per-provider throttle permit and runs analyzeWithAgent.
  • Costs are bumped on Report.iconCost regardless of success; the matching IconCandidate flips to Done (with the resolved emoji) or Error.

The user taps any returned emoji to commit it to:

  • The report (alternative icons from the Hub) — replaces Report.icon + clears iconWinningTier.
  • One agent only (alternative icons from the per-agent detail) — replaces ReportAgent.icon + clears iconWinningTier so the per-agent prompt card stops claiming a tier produced the icon.

Lives in AppViewModel.iconFanOutByReport: Map<reportId, List<IconCandidate>> — in-memory only. AlternativeIconsScreen reads it via collectAsState; navigating away mid-flight preserves the in-flight list (the outer appViewModel.viewModelScope.launch keeps running). A Restart button on the screen re-fires every pick that's not already in a terminal state.

The icons grid

Top-bar Create action on the report result screen → View → Icons. Minimal grid of every successful per-agent icon, plus a top-left thumb of the per-report icon. Tapping a glyph opens that agent's Model response detail; backing out returns to the grid (not the result screen — overlay back-stack rule).

Help topic: report_icons_grid. Grid spacing adapts down when not every icon fits at the default size, so a 12-agent report still lays out without wrapping.

Costs surface

Per-icon spend surfaces in three places:

  1. In-app View → Costs — By type / By model / All tabs. The per-agent chain calls appear under each agent's provider / model split with a Calls column showing the number of tier attempts; the per-report icon appears as its own row.
  2. HTML export — Complete and Costs views — same three tabs, same per-call All rendering. The icon row in the Complete view shows the agent's effective icon + cost.
  3. Global Usage statistics — each tier call posts with kind="icon", attributed to the model that actually ran. Filterable in the Usage screen.

Report.costsFromDeletedItems (also bumped on per-agent / secondary / fan-out deletions) surfaces as its own line above the Total when non-zero — so the user sees what the API actually billed for the run, even after trimming rows.

Fan-icons (parallel feature)

A separate emoji chain runs on every Fan-out pair (not on report agents), driven by FanOutEngine. Same 3-tier shape (tier 1: chat continuation; tier 2: one-shot internal/report_icon; tier 3: internal/report_icon_3th against the DeepSeek-pinned agent) and same emoji-normalisation rules. Lives in FanOutIconsScreens.kt and routed through FanOutMode.ICONS. The batch:

  • Runs separately from the parent fan-out, gated by maxConcurrentFanIconsCalls so it doesn't starve the parent's budget.
  • Stamps PairState.icon, iconWinningTier, iconInputCost, iconOutputCost, iconErrorMessage so the L1 stats and Costs view reflect both the pair and its icon side-by-side. Costs surface on a dedicated fan-icons row in the report Costs view.
  • Auto-resumes on screen entry if the user closed the report mid-batch. A 429 on a chain tier benches the model in ModelCooldownStore; subsequent pairs skip the benched (provider, model).
  • Has its own Clear fan-icons action that wipes icon state without deleting pairs.
