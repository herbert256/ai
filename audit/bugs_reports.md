## File: ai/src/main/java/com/ai/ui/report/ReportScreen.kt

### Bug 1 ‚Äî Severity: HIGH ‚Äî Category: State / recomposition
**Location:** lines 1672‚Äì1676 (`GenerationPhase`, `agentCost`)
**Symptom:** `agentCost` is recomputed on EVERY recomposition (no `remember`), and during a live batch the surrounding screen recomposes many times per second (running cross pairs, batch tick, etc.). Each recomposition walks every entry in `reportsAgentResults`, calls `PricingCache.getPricing` (disk-backed) and `resolveModelForResult` for each. This is justified in the comment as "needed to pick up DEFAULT_PRICING", but the comment ignores that `PricingCache.getPricing` is supposed to be cheap-read; the heavy work is the ModelType + Pricing lookups. On a 30-agent report with batching this is hundreds of recomputations/second.
**Root cause:** Memoization removed deliberately, but the explanation is wrong ‚Äî the cold-cache window is short and `PricingCache.preloadCompleted` could be used as a remember key to refresh once.
**Reproduction:** Open a 30-agent report and start a long meta batch. Profile CPU.
**Proposed fix:** `remember(reportsAgentResults, PricingCache.preloadCompleted)` for `agentCost`.
**Status:** Open

### Bug 2 ‚Äî Severity: HIGH ‚Äî Category: State / recomposition
**Location:** lines 1789‚Äì1796 (`GenerationPhase` scroll-to-top loop)
**Symptom:** When the user opens a report, `LaunchedEffect(currentReportId)` fires a 15-iteration loop that calls `resultListState.scrollToItem(0)` every 100ms. This forcibly resets the LazyColumn scroll position 15 times in the first 1.5s ‚Äî meaning if the user starts scrolling immediately upon opening (e.g. swipe-up gesture for a long report), their scroll input is repeatedly clobbered.
**Root cause:** The fix for "secondary rows arriving and scrolling agent rows below the fold" was implemented as a brute-force loop instead of a one-shot reaction to the secondary rows arriving.
**Reproduction:** Open a report with many cross summaries, immediately swipe up. The list snaps back ~15 times.
**Proposed fix:** Trigger one scroll-to-top on `LaunchedEffect(currentReportId)`, then a second one keyed on the size growth of secondary/cross/translation lists (or do nothing ‚Äî Compose will leave scroll position alone).
**Status:** Fixed (this session) ‚Äî drop the 15 √ó 100ms scroll loop; one-shot scroll on report open

### Bug 3 ‚Äî Severity: MEDIUM ‚Äî Category: Concurrency
**Location:** line 162 (`ReportsScreenNav` translation runs `collectAsState`)
**Symptom:** `reportViewModel.translationRuns.collectAsState().value.values.toList()` is called on every recomposition. `collectAsState` returns a State, but `.value.values.toList()` allocates a fresh List on every recomposition. That fresh list is then passed by value into the nested `ReportsScreen` callable, which uses it as a `LaunchedEffect` key further down (`finishedSignature`), causing the LaunchedEffect to potentially re-key spuriously based on whether the underlying Map values are reference-equal between collections (they are, but the wrapping List isn't).
**Root cause:** `.toList()` on a StateFlow's values doesn't preserve structural identity for keys.
**Reproduction:** Profile recompositions during a translation run.
**Proposed fix:** Use `derivedStateOf` or pass the Map directly.
**Status:** Open

### Bug 4 ‚Äî Severity: HIGH ‚Äî Category: Wrong logic
**Location:** lines 540‚Äì548 (`Auto-generate for external models` LaunchedEffect)
**Symptom:** When external models arrive, the LaunchedEffect immediately calls `onGenerate(updatedModels, ‚Ä¶)` BEFORE the parallel `LaunchedEffect(externalModels)` on lines 551‚Äì555 has a chance to update `models` in state. ### Bug 6 ‚Äî Severity: MEDIUM ‚Äî Category: Stale data / state
